## C++ Interaction System Summary

The interaction system is built in C++ at `Source/SohVr/` with these key classes:

| Class | Role |
|---|---|
| `ActivatableActor` | Base class for actors that can be enabled/disabled via `EnablerComponent` |
| `EnablerComponent` | Listens for `Message.Event.StoryStep` to enable/disable its owning actor based on step matching |
| `TeleportPoint` | Extends `ActivatableActor`, fires `OnPlayerTeleported` delegate when used |
| `SceneManager` | Game instance subsystem managing scene loading/unloading/transitions |
| `StoryHelpers` | Blueprint function library for story steps, teleportation, data table loading |

## Data Flow

The core data flow relies on the Gameplay Message Subsystem:

- **`Message.Event.StoryStep`**: This channel is used to broadcast integer step numbers. `EnablerComponent` listens to this channel to activate/deactivate actors.
- **`Message.Event.TeleportPoint`**: This channel is used to broadcast information about teleport point usage.

## Story Step, Level Sequence, and Script Connection

### The Three Pillars

1. **The Script (OC Screenplay)**: This is the narrative source of truth. It defines the scenes, the locations, and the key moments or interactions that must happen to advance the story. The `interaction_map.md` document translates the script's beats into concrete VR interactions (e.g., "GRIP - Grab DOOR KNOB").

2. **Story Steps (`DT_StorySteps`)**: This is a Data Table in Unreal Engine that acts as the bridge between the script and the game logic. Each row in this table represents a specific moment or beat in the story. The `VERIFIED_STEP_SEQUENCE_MAP.txt` document provides the ground truth mapping from the Data Table's row index to the corresponding Level Sequence.

3. **Level Sequences (`LS_...`)**: These are cinematic sequences created in Unreal Engine that play out the narrative moments defined in the script — character animations, camera movements, dialogue, and visual effects. Each Level Sequence corresponds to a specific Story Step.

### How They Connect: The Data Flow

The connection between these three pillars is managed by the Gameplay Message Subsystem and the custom C++ components:

1. **Player Interaction**: The player performs an action in the world, such as grabbing an object (`GrabbableComponent`) or teleporting to a location (`TeleportPoint`).

2. **Blueprint Logic**: This interaction triggers a Custom Event in the Level Blueprint (e.g., `OnGripDoorKnob`).

3. **Broadcast Story Step**: The Blueprint logic then uses the `K2_BroadcastMessage` function to send a message on the `Message.Event.StoryStep` channel. The payload of this message is a `Msg_StoryStep` struct containing an integer `Step` value. This integer corresponds to a row index in the `DT_StorySteps` Data Table.

4. **EnablerComponent Listens**: The `EnablerComponent` (which is attached to actors like `LevelSequenceActor` and `TeleportPoint`) is constantly listening to the `Message.Event.StoryStep` channel.

5. **Trigger Level Sequence**: When an `EnablerComponent` hears a `Step` number that matches one of its `EnableOnStorySteps` values, it enables its owning actor. If the owning actor is a `LevelSequenceActor`, it begins to play its Level Sequence.

This creates a clear, data-driven chain of events:

`Player Action -> Blueprint Event -> Broadcast Step # -> EnablerComponent Hears Step # -> Play Level Sequence`

This allows the narrative to be controlled and sequenced directly from the `DT_StorySteps` Data Table, making it easy to reorder events or change which Level Sequence plays for a given story beat without having to rewire the Level Blueprints.

### Example: Scene 3 - Grabbing the Door Knob

- **Script Beat**: Heather and her friends enter the house.
- **Interaction Map**: `GRIP - Grab DOOR KNOB`
- **Story Step Map**: `DoorKnob grabbed -> Step 7 (LS_3_1)`
- **Data Flow**:
    1. Player grips the door knob.
    2. The `GrabbableComponent` on the door knob fires its `OnGrabbed` event.
    3. This triggers the `OnGripDoorKnob` Custom Event in `SL_Trailer_Logic`.
    4. The Blueprint broadcasts `Step = 7` on the `Message.Event.StoryStep` channel.
    5. The `EnablerComponent` on the `LS_3_1` Level Sequence Actor hears `Step = 7` and plays the sequence, which shows Heather and her friends entering.

## Interaction Chain Pattern

Every interaction chain follows this standard pattern (with minor variations):

```
Comment Node (describes the chain)
  |
CustomEvent (triggered by player interaction)
  |
DisableActor (disable the current interactable)
  |
GetSubsystem (GameplayMessageSubsystem)
  |
BroadcastMessage (Message.Event.StoryStep, step=N)
  |
EnableActor_Next (enable the next interactable) [omitted on last chain in scene]
  |
[Optional: SpawnActor for next scene's teleport point]
```

Variations:
- **Scene5_HandPlaced** includes `PlayHapticEffect` (heartbeat) before DisableActor
- **Scene transition chains** (last chain in a scene) include `SpawnActor` instead of `EnableActor_Next`
- **Scene6_ComputerActivated** and **Scene3_DoorKnobGrabbed** include `SpawnActor` nodes

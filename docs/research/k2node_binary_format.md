# Unreal Engine K2Node Extras Blob Binary Format

**Author:** Manus AI
**Date:** 2026-03-05
**Version:** 2.0

## 1. Overview

This document provides a comprehensive technical breakdown of the binary serialization format for the `Extras` blob found in `K2Node` exports within Unreal Engine `.uasset` and `.umap` files. This format is used to store the structure of the Blueprint graph, including nodes, pins, and their connections.

The information herein has been verified against Unreal Engine 5.6 and confirmed to be stable and backward-compatible with versions 5.4 and 5.5 [1]. The primary focus is on the data serialized by `UEdGraphNode::Serialize()` and its constituent parts, which form the core of the `Extras` blob for all `K2Node` types.

Understanding this format is critical for any tool that aims to programmatically read, create, or modify Blueprint graphs at a binary level, bypassing the Unreal Editor.

## 2. UEdGraphNode::Serialize and the Pin Array

The `Extras` blob for a `K2Node` begins with the data serialized by its parent class, `UEdGraphNode`. The most crucial part of this is the pin array, which defines all the input and output pins of the node.

The structure is as follows:

| Field         | Data Type | Description                                      |
|---------------|-----------|--------------------------------------------------|
| **NumPins**   | `int32`   | The total number of pins serialized for this node. |
| **Pin Array** | `Pin[]`   | An array of `NumPins` serialized pin structures.   |

Following this pin array, certain subclasses may write additional data. For example, nodes inheriting from `UK2Node_EditablePinBase` will serialize a `TArray<FUserPinInfo>` [1].

## 3. UEdGraphPin::Serialize

Each pin in the pin array is serialized by the `UEdGraphPin::Serialize()` method. This method writes a sequence of fields that describe the pin's identity, properties, and connections.

The exact field order is critical and has been confirmed as stable [1].

| Field                       | Data Type                | Description                                                                  |
|-----------------------------|--------------------------|------------------------------------------------------------------------------|
| **OwningNode**              | `FPackageIndex`          | A reference to the export that owns this pin (i.e., the node itself).        |
| **PinId**                   | `FGuid`                  | A unique identifier for this pin within the entire Blueprint.                |
| **PinName**                 | `FName`                  | The internal name of the pin (e.g., `"execute"`, `"then"`).                  |
| **PinFriendlyName**         | `FText`                  | The display name of the pin shown in the editor.                             |
| **SourceIndex**             | `int32`                  | Internal index, often 0.                                                     |
| **PinToolTip**              | `FString`                | The tooltip text displayed when hovering over the pin.                       |
| **Direction**               | `uint8`                  | `0` for Input (`EGPD_Input`), `1` for Output (`EGPD_Output`).                |
| **PinType**                 | `FEdGraphPinType`        | A complex structure defining the pin's data type (see section 4).            |
| **DefaultValue**            | `FString`                | The default value of the pin, represented as a string.                       |
| **AutogeneratedDefaultValue** | `FString`                | The autogenerated default value, if any.                                     |
| **DefaultObject**           | `FPackageIndex`          | A reference to a default object, if applicable.                              |
| **DefaultTextValue**        | `FText`                  | The default value as an `FText` object.                                      |
| **LinkedTo**                | `TArray<UEdGraphPin*>`   | An array of pointers to other pins this pin is connected to.                 |
| **SubPins**                 | `TArray<UEdGraphPin*>`   | An array of pointers to any sub-pins (e.g., for expanded structs).           |
| **ParentPin**               | `UEdGraphPin*`           | A pointer to the parent pin, if this is a sub-pin.                           |
| **RefPassThrough**          | `UEdGraphPin*`           | A pointer used for reference pass-through.                                   |
| **PersistentGuid**          | `FGuid`                  | A GUID that persists across compilations.                                    |
| **BitField**                | `uint32`                 | A bitfield containing various boolean flags.                                 |

## 4. FEdGraphPinType::Serialize

The `FEdGraphPinType` structure is one of the most complex parts of the pin serialization. It defines the pin's data type, such as `exec`, `bool`, `object`, `struct`, etc. Its serialization order is also stable and confirmed [1].

| Field                      | Data Type                | Description                                                                  |
|----------------------------|--------------------------|------------------------------------------------------------------------------|
| **PinCategory**            | `FName`                  | The primary category (e.g., `"exec"`, `"struct"`, `"object"`).             |
| **PinSubCategory**         | `FName`                  | The sub-category, often providing more specific type info.                   |
| **PinSubCategoryObject**   | `FPackageIndex`          | A reference to the class or struct object for this type.                     |
| **ContainerType**          | `uint8`                  | `0` for None, `1` for Array, `2` for Set, `3` for Map Key, `4` for Map Value.  |
| *PinValueType*             | *(struct)*               | *(Only if ContainerType is Map)* A nested struct defining the map's value type. |
| **bIsReference**           | `int32` (as bool)        | Is this a pass-by-reference pin?                                             |
| **bIsWeakPointer**         | `int32` (as bool)        | Is this a weak object pointer?                                               |
| **MemberParent**           | `FPackageIndex`          | (FSimpleMemberReference) Reference to the parent class of the member.        |
| **MemberName**             | `FName`                  | (FSimpleMemberReference) Name of the member variable or function.            |
| **MemberGuid**             | `FGuid`                  | (FSimpleMemberReference) GUID of the member.                                 |
| **bIsConst**               | `int32` (as bool)        | Is this a const pin?                                                         |
| **bIsUObjectWrapper**      | `int32` (as bool)        | Is this a UObject wrapper?                                                   |
| **bSerializeAsSingle...**  | `int32` (as bool)        | Should this be serialized as a single-precision float?                       |

## 5. Tagged Properties vs. Extras Blob

It is crucial to distinguish between data stored in the `Extras` blob and data stored as **Tagged Properties** directly on the export. While the `Extras` blob contains the graph structure (pins and links), fundamental node properties are stored in the export's `Data` array.

Common tagged properties for K2Nodes include:

- **NodePosX**, **NodePosY**: The node's position in the graph editor.
- **NodeGuid**: A unique GUID for the node itself.
- **CustomFunctionName**: For `K2Node_CustomEvent`, this stores the event's name.
- **FunctionReference**: For `K2Node_CallFunction`, this stores a reference to the function being called.

Any robust binary editor must read and write both the `Extras` blob and these tagged properties correctly.

## 6. ScriptSerializationEndOffset (SSEO)

Every export in a `.umap` file has a `ScriptSerializationEndOffset` (SSEO) field stored in the export table. This field marks the byte offset from the start of the export's serialized data to the point where tagged property serialization ends (just past the `None` terminator). The engine uses this as a boundary check during deserialization.

**Critical finding:** UAssetAPI correctly recalculates `SerialSize` when export data changes, but does NOT recalculate SSEO. If you modify tagged properties in a way that changes their serialized size (adding/removing properties, changing variable-length string values), SSEO becomes stale and the engine will assert:

> `ScriptSerializationEndOffset X does not match offset during deserialization Y for object Z`

For K2Node exports specifically, the relationship is:

```
SSEO = SerialSize - ExtrasSize - 4
```

Where `ExtrasSize` is the byte length of the Extras blob and the `4` accounts for the trailing int32 after the Extras blob.

This formula does NOT hold for all export types. `BlueprintGeneratedClass`, `Level`, and CDO exports have additional serialization data between the SSEO boundary and the Extras blob.

### Export Table Entry Layout (UE 5.6)

The export table entry is 112 bytes per entry. Key fields:

| Field Offset | Data Type       | Field Name                      |
|-------------|-----------------|----------------------------------|
| +0          | `FPackageIndex` | ClassIndex                       |
| +4          | `FPackageIndex` | SuperIndex                       |
| +8          | `FPackageIndex` | TemplateIndex                    |
| +12         | `FPackageIndex` | OuterIndex                       |
| +16         | `FName`         | ObjectName                       |
| +24         | `int64`         | SerialSize                       |
| +32         | `int64`         | SerialOffset                     |
| ...         | ...             | (other fields)                   |
| +100        | `int64`         | ScriptSerializationEndOffset     |

This layout has been verified against the Oculus fork of UE 5.6 and confirmed by Epic's Developer Assistant [2].

### Safe vs. Unsafe Modifications

| Modification Type                          | SSEO Impact | Patcher Needed? |
|-------------------------------------------|-------------|------------------|
| Extras-only (pin wiring, default values)  | None        | No               |
| Fixed-size property value change (enum)   | None        | No               |
| Variable-length property change (strings) | SSEO stale  | Yes              |
| Property addition or removal              | SSEO stale  | Yes              |

## 7. Package Trailer Offset

UE 5.6 packages include a Package Trailer at the end of the file, identified by the magic bytes `0xC1832A9E` (little-endian for `0x9E2A83C1`). The offset to this trailer is stored in the package summary header. When export data grows or shrinks, the trailer moves but the header offset is not always updated by UAssetAPI.

The post-save patcher must scan for the trailer magic and update the header offset accordingly.

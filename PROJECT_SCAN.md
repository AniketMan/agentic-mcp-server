## Level Architecture

The project uses a **World Composition / Sublevel Streaming** pattern:

- **ML_Trailer** is the **persistent master level**. It contains only infrastructure (WorldSettings, NavMesh, GPULightmass, and the `ML_Trailer_C` level script actor). It loads/unloads the SL_ sublevels via `SceneManager`.
- **SL_ sublevels** are streamed in/out by the `SceneManager` subsystem. Each SL_ handles one concern:
  - `SL_Trailer_Art` — static meshes (59 actors, all StaticMeshActor)
  - `SL_Trailer_Blockout` — greybox geometry (3 actors)
  - `SL_Trailer_Lighting` — lights and post-process (28 actors: spots, points, directional, sky, post-process volume)
  - `SL_Trailer_Logic` — **all interactive actors and Blueprint logic** (22 actors, 7 functions, 165 exports)

## Data Flow

```
DA_GameData (UGameData)
  -> ScenesSequence: array of FPlayableScene (SceneName + SceneAsset)
  -> SceneSequenceOverride actor overrides this at BeginPlay

DT_StorySteps (DataTable of FStoryStep rows)
  -> Each row: SceneIndex, LevelName, LevelSequence (soft ref), AutoAdvance, CanSkip, StartTime, PlayRate, IsLoop, AllowTeleport

Message System (GameplayMessageRuntime):
  -> Channel: Message.Event.TeleportPoint  |  Payload: Msg_TeleportPoint (contains soft ref to TeleportPoint actor)
  -> Channel: Message.Event.StoryStep      |  Payload: Msg_StoryStep (contains Step integer)

EnablerComponent listens for Message.Event.StoryStep and enables/disables its owning actor based on step matching.
```
